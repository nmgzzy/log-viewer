# 桌面日志客户端 —— 需求文档（PRD）

本客户端是 `uf_log` 日志系统的**上位机/查看端**：在 PC 上**实时接收**嵌入式设备经 UDP 发来的日志，
并**打开/分析**本地导出的日志文件，提供过滤、格式化、高亮、多标签、合并、仪表盘等能力。
与设备侧设计见 `[logging-system.md](logging-system.md)`、落地见 `[logging-implementation-notes.md](logging-implementation-notes.md)`。

> 优先级标记：**必须**（MUST，MVP 必备）/ **应**（SHOULD，重要）/ **可**（MAY，增强）。
> 容量基线：**单个工作集 < 1,000,000 行**，在此规模内所有交互须保持流畅（见 §7 性能与容量）。

---

## 1. 背景与定位

设备侧 `uf_log` 把每条日志组装为标准 **RFC5424**，经 `/dev/log` 交 syslog-ng：
①落本地文件 `/var/log/uf/messages`（logrotate 压缩/循环）；②经 **UDP** 转发到上位机。
本客户端就是 ② 的接收端，同时也能离线打开 ① 导出的文件。

**与现有标准工具的关系**：因设备发的是标准 RFC5424，理论上任何 syslog 服务器都能收；但它们不便于
**实时浏览、按 tag/host 多维过滤、高亮、合并多设备、轻量仪表盘**。本客户端补齐这块"开发/调试/现场
排障"的交互体验，且**不仅限于 uf_log**——能兼容通用 syslog / JSON 行日志源。

---

## 2. 目标与非目标

**目标**

- 跨平台、高性能、轻量化、可扩展的**桌面日志查看与分析**客户端。
- 一处同时看：本地文件 + 多设备网络流，并能**合并成统一时间线**。
- 让开发/测试/现场人员**秒级定位**问题（过滤/搜索/高亮/仪表盘）。

**非目标（明确排除）**

- 不做长期海量日志仓库/检索后端（那是 ELK/Loki 等的领域）；本客户端面向**单工作集 < 百万行**的
即时分析。
- 不做设备端采集（那是 `uf_log` + syslog-ng）。
- 不做告警编排/工单系统（仅做仪表盘内的轻量阈值高亮，见 §6 FR-9）。
- 首版不做多人协作/云端同步。

## 3. 统一日志数据模型（与 uf_log 对齐）

所有输入源解析后归一为同一条记录（client 内部 canonical record）：


| 字段         | 类型        | 来源（uf_log）                     | 说明                                      |
| ---------- | --------- | ------------------------------ | --------------------------------------- |
| `ts`       | 时间戳(含毫秒)  | RFC5424 TIMESTAMP / 文件 ISODATE | 排序与合并主键；保留原始时区                          |
| `host`     | string    | HOSTNAME                       | 设备标识（多设备区分）                             |
| `app`      | string    | APP-NAME（进程名）                  | 产生进程名                                   |
| `pid`      | int?      | PROCID                         | 可空（内核日志为空：`kernel[]`）                   |
| `level`    | enum(0–7) | PRI 低 3 位 / 文件 LEVEL           | syslog 8 级；常用 err/warning/info/debug    |
| `facility` | enum      | PRI 高位                         | 区分应用(local0)/内核(kern)等                  |
| `tag`      | string    | MSGID                          | 业务模块标签（多标签过滤/着色的核心维度）                   |
| `msg`      | string    | MSG                            | 正文（已在设备端净化为单行）                          |
| `source`   | string    | —                              | 该记录来自哪个源（文件路径 / `udp:host:port`），用于合并标注 |
| `raw`      | string    | —                              | 原始行，留作"显示原文/解析失败回退"                     |


> 设备侧两种实际格式客户端都须认（见 §5）：
>
> - **UDP（RFC5424）**：`<134>1 2026-06-12T10:16:41.834Z host app 117 i2c - msg`
> - **本地文件（syslog-ng 模板）**：`2026-06-12T10:16:41.834+00:00 host app[117] info i2c: msg`

## 4. 输入源（数据来源）

- **FR-S1（必须）本地文件**：打开单个/多个文件或整个目录；支持设备导出的 `messages` 及其轮转产物
`messages.1`、`messages.2.gz`（**自动解压 .gz** 顺序拼接）。
- **FR-S2（必须）UDP 网络**：监听可配置端口（默认 **514**）；同时接收**多台设备**（按 `host` 区分）；
绑定地址可配（默认 `0.0.0.0`，可限定网卡）。对应设备侧 `syslog-ng.conf` 的 `network("<本机IP>" udp 514)`。
- **FR-S3（应）实时跟随（follow / tail -f）**：本地文件追加与轮转切换时无缝跟随，不丢行不重读。
- **FR-S4（可）其它源**：TCP（含 TLS）syslog、标准输入/管道、经 SSH 远程 `tail`。
- **可靠性**：UDP 接收用**有界缓冲 + 背压**；突发超载时按策略丢弃并**显式计数提示**（不静默丢）。

---

## 5. 解析与格式兼容

- **FR-P1（必须）RFC5424**：完整解析 PRI/版本/TIMESTAMP(含小数秒)/HOSTNAME/APP-NAME/PROCID/MSGID/
STRUCTURED-DATA/MSG。
- **FR-P2（必须）uf_log 文件模板**：`ISODATE HOST APP[PID] LEVEL TAG: MSG`，须容忍空 PID（`app[]`）。
- **FR-P3（应）RFC3164（传统 syslog）**：兼容老格式（无毫秒、无独立 tag）。
- **FR-P4（应）JSON 行**：每行一个 JSON（设备侧可把 UDP destination 模板改为 JSON）；字段名可映射配置。
- **FR-P5（必须）健壮性**：解析失败的行**不丢弃**，以 `raw` 原样展示并标记"未解析"；混合格式自动探测。
- **FR-P6（可，扩展点）自定义解析器**：用户可按正则/字段映射定义新格式（见 §8 可扩展性）。

## 6. 功能需求

### FR-1（必须）本地日志查看

打开文件/目录 → 解析 → 虚拟化列表展示；支持跟随、跳转行号/时间、定位首/尾、按列展示。

### FR-2（必须）网络日志查看

启动 UDP 监听即实时显示来流；每条标注来源 `host`；可暂停/继续、清空、限流提示；可同时多端口/多源。

### FR-3（必须）存储

- 接收/打开的日志进入**本地存储**（内存索引 + 可选落盘缓存），工作集 < 百万行须全程可交互。
- **网络日志自动落盘归档（必须，已确认）**：UDP/网络收到的日志**自动持久化**到本地归档文件
  （按 `host` 分文件或统一文件可配），带**轮转**（按大小/时长）与保留份数；程序重启后可重新打开归档继续分析。
  归档格式默认与设备侧一致（RFC5424 或 uf_log 文本行），便于互通。
- **保留策略**：内存工作集环形上限（按行数/字节/时长），超限滚动丢最旧并提示；**落盘归档不受此上限影响**
  （归档自有轮转），即"内存里看最近的，磁盘上留全的"。
- **导出**：把当前/过滤后的结果导出为 文本 / JSON / CSV（FR-11）。
- **可**：会话持久化（关闭后重开恢复上次工作集与视图）。

### FR-4（必须）过滤

- 维度：`level`（≥某级别）、`tag`（**多选**）、`host`、`app`、`pid`、**时间范围**、`msg` 全文/正则。
- 组合：包含/排除（include/exclude），多条件 AND/OR；一键快速过滤（点某字段值即过滤）。
- **保存的过滤器**（命名、复用、导入导出）；过滤**实时**作用于已加载与新进流。
- 性能：百万行上过滤结果 **应 ≤ ~1s**（见 §7）。

### FR-5（必须）格式化与显示

- 列式视图，列可选/可排序/可调宽（ts/host/app/pid/level/tag/msg）；紧凑与详情两种密度。
- 时间显示：绝对/相对（相对首行或上一行 Δt）、本地时区/原始时区切换、毫秒可见。
- 长消息换行/截断切换；单元格复制、整行复制（原文或字段化）。

### FR-6（必须）高亮

- 按 `level` 的**默认配色**（err 红 / warning 黄 / info 常规 / debug 灰等）。
- **规则高亮**：按关键字/正则/字段条件设定前景背景色与样式；规则可排序、可启停、可导入导出（规则包）。
- 命中搜索词高亮；当前行定位高亮。

### FR-7（必须）多标签 = 多文件/多源浏览（标签页）

**已确认**："多标签"指**同时浏览多个 log 文件/源**。采用 **MDI 标签页**（每个打开的文件/源 = 一个 Tab）：

- 同时打开多个本地文件（及目录展开的多文件）与多个网络源，每个占一个 Tab，标题显示来源；可快速切换。
- 每个 Tab **独立保有**自己的过滤、搜索、高亮、列布局、滚动位置；互不干扰。
- **可**：Tab 拖拽分屏并排对比；关闭/复制 Tab；"固定/钉住"常看的源。
- 与 **FR-8 合并** 配合：可把选中的多个 Tab 合成一个"合并时间线" Tab（见 FR-8）。

> 说明：按 `tag` 的多选过滤/着色/分面统计属于**过滤维度**，并入 FR-4 / FR-9，不在本条。

### FR-8（必须）合并日志

- 将多个源（多文件 + 多设备 UDP 流）**按 `ts` 合并为统一时间线**，每行标注 `source`/`host`。
- 乱序到达的网络日志按时间窗重排序；**可**按字段去重（合并重复转发）。

### FR-9（应）简单仪表盘

- 实时小面板：按时间桶的**总量/各 level 计数曲线**、错误率、**Top tag / Top app / Top host**、当前接收速率。
- 轻量**阈值高亮**：如"错误速率 > 阈值"变红；点图元可下钻回过滤后的列表。
- 面向"一眼看健康度"，**非**重型 BI。

### FR-10（应）搜索

全文/正则快速搜索，结果计数、上一处/下一处跳转、搜索即高亮；与过滤解耦（搜索不改变可见集）。

### FR-11（应）导出/分享

导出当前过滤结果为 文本/JSON/CSV；**可**生成可分享的"问题片段"（带上下文 N 行）。

### FR-12（必须）syslog / JSON 兼容

不仅认 uf_log，也能直接作为通用 RFC5424/RFC3164/JSON-行 的查看端（FR-P1~P4）。

## 7. 非功能需求

### 7.1 性能与容量（核心约束：单工作集 < 100 万行）

- **冷加载**：百万行文件（含 .gz 解压）**应**在数秒内可浏览（流式加载，边读边显示）。
- **滚动**：虚拟化渲染，任意规模下**必须** ≈ 60fps，内存不随可见区外行数线性膨胀。
- **过滤/搜索**：百万行**应** ≤ ~1s 出结果（建立轻量列式索引 / 列存）。
- **UDP 接收**：持续 **应** ≥ 50,000 行/秒 不丢（有界队列 + 背压）；突发按策略丢并计数。
- **内存**：百万行常驻**应** ≤ ~300–500 MB（紧凑记录表示 + 字符串驻留/分块）。
- **空闲占用**：极低 CPU/内存（事件驱动，无忙轮询）。

### 7.2 跨平台

- **必须**：Windows 10+、macOS 12+、Linux（x86_64 与 arm64）。
- **应**：单一自包含可执行文件/安装包，无需用户额外安装重运行时；安装体积尽量小（轻量化）。

### 7.3 可靠性

- 解析失败、格式混杂、超长行、乱序、源断连/重连，均**不得崩溃**；以可见方式降级。
- 长时间运行（数天）无内存泄漏/无限增长（受保留策略约束）。

### 7.4 安全

- UDP 监听默认仅本机网段可达；绑定地址/端口可配；**必须**校验输入边界，**日志内容绝不被当作代码/命令执行**。
- 高亮/解析的正则须防 ReDoS（超时保护）。

### 7.5 易用性 / 国际化

- 中文/英文界面切换；深/浅色主题；键盘优先操作（搜索、过滤、跳转快捷键）。

## 8. 可扩展性（架构性要求）

以"核心 + 插件点"组织，便于后续拓展而不动核心：

- **源插件**：file / udp / tcp(tls) / stdin / ssh-tail …，统一产出"行流"。
- **解析器插件**：RFC5424 / RFC3164 / uf_log 模板 / JSON / 用户自定义（正则+字段映射）。
- **高亮规则包**：可导入导出、可分享。
- **导出器**：text / json / csv …可扩展。
- **可（远期）**：脚本化处理钩子（对每行做转换/打标）、自定义仪表盘部件。

## 9. 与 uf_log 系统的对接细节（集成约定）

- **UDP**：设备侧 `syslog-ng.conf` 的 `destination d_net { network("<上位机IP>" transport("udp") port(514) flags(syslog-protocol)); }`——即设备主动发往**上位机 IP:514**（RFC5424）。客户端在该端口监听即可。
  - QEMU 联调：guest 经 user-net 发 `10.0.2.2`（主机）；或用 `run-qemu.sh -p` 端口转发。
- **本地文件**：设备上为 `/var/log/uf/messages` + 轮转 `messages.1` / `messages.{2..N}.gz`；
导出到 PC 后用 FR-S1 打开（多文件按序、.gz 自动解压、合并成一条时间线）。
- **字段映射**：见 §3 表；level 用 syslog severity 名（err/warning/notice/info/debug…），
tag 取 MSGID，app 取 APP-NAME，host 取 HOSTNAME。
- **JSON 模式（可选）**：若设备侧把 `d_net` 模板改为 JSON 行，客户端用 FR-P4 解析；字段名映射可配。
- **时区**：设备 TIMESTAMP 带时区偏移；客户端按 §FR-5 可在原始/本地时区间切换显示。

## 10. 里程碑（MoSCoW / 分期）

**MVP（必须）**：FR-S1/S2、FR-P1/P2/P5、FR-1、FR-2、FR-3（基础存储+保留 + **网络日志自动落盘归档**）、
FR-4（过滤）、FR-5（显示）、FR-6（高亮）、FR-7（多文件标签页）、FR-8（合并）、FR-12；
跨平台三端、中英文双语、性能与容量达标（§7.1）。

**R2（应）**：FR-S3 跟随、FR-9 仪表盘、FR-10 搜索增强、FR-11 导出、FR-P3/P4（RFC3164/JSON）、会话持久化。

**R3（可）**：FR-S4（TCP/TLS、SSH-tail）、自定义解析器/脚本钩子、自定义仪表盘、规则包市场/分享。

## 11. 技术选型建议（非约束，供决策参考）

需求是"跨平台 + 高性能 + 轻量 + 可扩展"，桌面 GUI 几条路线的取舍：


| 方案                                 | 优点                           | 代价                         | 适配度                     |
| ---------------------------------- | ---------------------------- | -------------------------- | ----------------------- |
| **Rust + egui/`iced`（即时模式原生 GUI）** | 高性能、单二进制极轻、内存可控、虚拟化列表好做、无 GC | 复杂 UI 控件需自绘；生态较新           | **高**（最契合高性能+轻量+百万行虚拟化） |
| **Rust/C++ + Qt**                  | 控件成熟、表格/虚拟视图强、三端稳            | 体积较大、授权/打包略重               | 高（重 UI 友好）              |
| **Tauri（Rust 后端 + Web 前端）**        | UI 开发快、跨平台体积比 Electron 小很多   | WebView 渲染百万行需自做虚拟化、性能略逊原生 | 中高                      |
| **Go + Fyne/Wails**                | 开发快、跨平台                      | 大数据虚拟化与内存控制偏弱              | 中                       |
| Electron                           | 生态最丰富                        | **体积/内存大**，与"轻量"冲突         | 低（不建议）                  |


**已确认（技术栈）**：采用 **Rust**。GUI 以"高性能+轻量+百万行虚拟化"为第一约束，**推荐 Rust 原生
即时模式 GUI（egui / iced）**——单二进制、内存可控、自绘虚拟化列表最契合本需求；若后续更看重成熟表格
控件，可改 **Rust + Qt 绑定**。界面**必须中英文双语**（见 §7.5）。

## 12. 已确认的关键决策

| # | 议题 | 结论 | 对设计的影响 |
|---|------|------|--------------|
| 1 | "多标签"含义 | **同时浏览多个 log 文件/源**，用 MDI 标签页（每源一 Tab，独立过滤/高亮） | 见 **FR-7**；tag 维度归 FR-4/FR-9 |
| 2 | 日志如何到 PC | **UDP 实时 与 文件离线分析两者都要** | FR-S1 与 FR-S2 同为 MVP 必备 |
| 3 | 同时接收设备数 | **个位数**（不会太多） | 合并/仪表盘按小规模优化即可，无需分布式；合并排序、按 host 分面都很轻松 |
| 4 | 网络日志落盘 | **要**：自动归档 + 轮转，重启可重开 | 见 **FR-3** 网络日志自动落盘归档（MVP 必须） |
| 5 | 技术栈 | **Rust**（GUI 推荐 egui/iced），**中英文双语** | 见 §11、§7.5 |

### 仍可后续细化（不阻塞 MVP）
- 仪表盘深度：先做"健康度一眼看"（§FR-9），可下钻/时间序列分组留 R2 再评。
- JSON 模式字段名：默认沿用 §3 的 `ts/host/app/pid/level/tag/msg`，启用时若设备侧不同再做映射配置。
- 归档落盘的**分文件粒度**（按 host 一文件 vs 统一文件）与默认轮转参数：实现时给出默认值并可配。
